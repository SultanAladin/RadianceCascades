## Plugin pattern (load-bearing principle, reference)

Every swappable algorithm family has an abstract interface in `Source/<Family>/I<Family>Algorithm.h`:

* `IShadowAlgorithm` — PCF / PCSS / VSM / SDFCone implementors
* `IGIAlgorithm`     — SDFGI (only v1 implementor; voxel-cone / radiance-cascades drop in later)

`Renderer::Impl` holds `std::unique_ptr<IXxx>` per family + an enum + a setter. Switching algos = `Terminate()` old → `make_unique<NewImpl>()` → `Initialize()` new → rebuild lighting pipeline with new specialisation constant.

**Descriptor contract** (the load-bearing detail): the lighting compute shader reserves:
* `set=2, binding=0` for the shadow algo's output image (sampler2D / sampler2DShadow / R8 / RG32F depending on algo, with a spec-constant variant switch)
* `set=2, binding=1` for a 256-byte algo-private UBO
* `set=3, binding=0` for the GI algo's half-res output (RGBA16F)

Lighting never branches on `which algo`; the pipeline is recompiled on swap.

## Frame data flow (target architecture)

```
CameraView
   ▼
1. GBuffer pass        — raster GBuffer (albedo/normal/rough+metal+F0/emissive/depth/identity)
2. Shadow algo         — m_Shadow->RecordShadowPass(cmd, frame)    writes set=2,binding=0
3a. GI pre-frame       — m_GI->RecordPreFrame(cmd, frame)          probe relight (~512 / frame)
3b. GI gather          — m_GI->RecordGather  (cmd, frame)          writes set=3,binding=0
4. Lighting compose    — lighting.comp reads GBuffer + set2 + set3 + sun/sky/IBL → HDR RGBA16F
5. Tonemap (ACES)      — HDR → display gamut
6-8. Host: blit offscreen → swapchain → ImGui → present
```

### Acceptance criteria

* `RenderingSubsystem.exe` opens at 1280×720 with the grid still painting.
* ShaderBall visible at the world origin, flat-shaded (Lambert against a sun aimed roughly `(0.3, -0.8, 0.5)`); silhouette matches the reference image in the SolidArc copies.
* Camera orbit still works; rotating the camera shows the ShaderBall from every angle without z-fighting against the grid floor (the floor at Y=0 sits below ShaderBall's bottom AABB).
* New "Scene" panel (or extend "Camera"): triangle count + vertex count read from the live MeshRegistry.

### Risks for Phase 5

* OBJ parsers always look easy until they aren't — ShaderBall *is* clean (8.5 MB quad-dominant watertight), but n-gon polygons (`f a b c d e`) still need triangulation. Test on the file itself before generalising.
* Two vertex buffers vs interleaved layout — Vulkan pipeline state needs to know about both bindings. Stick to interleaved `struct { vec3 pos; vec3 normal; }` for Phase 5 simplicity; switch to SoA later if Phase 6 wants it.
* Depth attachment needs a barrier from UNDEFINED to DEPTH_STENCIL_ATTACHMENT_OPTIMAL the first time. Same per-swap-image framebuffer pattern as GridPass, plus a per-frame depth image (probably one shared depth target is fine since frames are serialised by `kFramesInFlight` fences).
* The ForwardPass is *throwaway* — keep it intentionally minimal so we're not tempted to grow features into it. Phase 6 (GBuffer) replaces it wholesale.
