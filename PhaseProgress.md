## Plugin pattern (load-bearing principle, reference)

Every swappable algorithm family has an abstract interface in `Source/<Family>/I<Family>Algorithm.h`:

* `IShadowAlgorithm` — PCF / PCSS / VSM / SDFCone implementors
* `IGIAlgorithm`     — Radiance Cascades × SHaRC hash is the only GI path. The `SDFGI` stub in `Source/GI/SDFGI.{h,cpp}` was the original v0 placeholder and is kept only because `IGIAlgorithm`'s plugin shape was designed to allow swapping — there is **no SDFGI implementation under active development**, and Phase 15 does not bring one back. If the SDFGI stub ever earns its keep, that's a Phase 16+ decision.

`Renderer::Impl` holds `std::unique_ptr<IXxx>` per family + an enum + a setter. Switching algos = `Terminate()` old → `make_unique<NewImpl>()` → `Initialize()` new → rebuild lighting pipeline with new specialisation constant.

**Descriptor contract** (the load-bearing detail): the lighting compute shader reserves:
* `set=2, binding=0` for the shadow algo's output image (sampler2D / sampler2DShadow / R8 / RG32F depending on algo, with a spec-constant variant switch)
* `set=2, binding=1` for a 256-byte algo-private UBO
* `set=3, binding=0` for the GI algo's half-res output (RGBA16F)

Lighting never branches on `which algo`; the pipeline is recompiled on swap.

- **Fix**: per-instance inverse-model SSBO at SDFCone set=2 binding 2. `Source/Renderer/InstanceXformBuffer.{h,cpp}` is a ring of `mat4 InverseModel[256]` host-visible SSBOs (one per frame-in-flight, 16 KB each), refreshed before each `RecordShadowPass`. `lighting_sdfcone.comp` reads `texelFetch(uIdentity, px, 0).r` for the instance id, fetches `invModel`, transforms `ro` and `rd` into mesh-local space, and scales `maxDistance`/`minStep` by `length(mat3(invModel) * L)` (= 100 for cm→m). The SDF AABB is already in mesh-local space so binding 1 needs no change.
- **All 9 grid balls** now shadow correctly off the single resident SDF. Multi-mesh SDFs (different per-instance SDFs) are still future work — same scope as Phase 14a's plan.
- Bumped baker resolution lid: combo now covers 32/64/96/128/256/512/1024 (was capped at 128). Warning banner appears at 256+ showing voxel count + VRAM cost (R16 = 2 GB at 1024^3).
- `-sdfcone` smoke run clean: zero Vulkan validation errors, SSBO binding writes succeed, descriptor pool sized for 3 types (CIS + UBO + SSBO).

### Phase 14a wrap-up notes

* **Plugin shape mirrors `IShadowAlgorithm`** — `IGIAlgorithm` has `Name()`, `Initialize(VulkanContext, GlobalSDF)`, `Terminate()`, `DrawImGuiParams()`, `RecordPreFrame(cmd, frame)`, `RecordGather(cmd, frame)`, `BindGIResourceForLighting(cmd, layout, set, binding)`. SDFGI ignores the pointer (its v0 path doesn't read GI settings), but the seam is in place for 14b.
* **Hot-swap is the same vkDeviceWaitIdle → Terminate → make_unique → Initialize → SetSettings pattern** as the shadow algos. `DrawRenderSettingsPanel` returns `-1` or a swap index, Main.cpp consumes it, the algo gets rebuilt. No descriptor-set thrash, no in-flight frame races.
* **`.gitignore` ignores all of `Docs/` including `Prototype/`.** Trade-off: the Taichi harness + viewer aren't tracked, so they won't survive a `git clean -fdx`. If you want to preserve them, they live in `C:\Users\OS\.claude\memory\project_rayarc_renderingsubsystem.md` linkages — pull them out manually before nuking the working copy. The decision was deliberate: design docs are ChatGPT exports + scratch HTML, and tracking them produces noisy diffs.
* **Taichi C4 (extension) PASS was on a small scene.** RMS=0.00 between extension-on and extension-off means c0 rays mostly terminated before extension fired — the math is consistent but the *stress* isn't there. The C++ implementation in 14b will exercise extension hard (large open scenes), so don't over-trust C4 as a green flag. If 14b shows weird darkening at cascade boundaries, suspect extension first.
* **Off-screen blind-spot is the load-bearing finding.** Screen-only insertion at 11.5% coverage is *not* enough — secondary insertion pass (camera frustum extruded by N cells, growing with cascade) is mandatory for production quality. Bake this into the 14b plan, don't ship "visible-pixel only" and patch later.
* **Camera-clipping fix in viewer is now permanent.** `enforce_camera_clearance(min_clear=6.0 cm)` runs after every `set_camera` call. Pushes camera backward along `f` until `sample_sdf(p) ≥ min_clear`. HUD shows `pushout` in cm so it's obvious when the camera is being relocated. Cap at 64 iterations / 1000 cm to prevent infinite loops in degenerate scenes.

### Phase 11 wrap-up notes

* **Set=2 contract now varies by variant.** PCF: 2 bindings (sampler2DArrayShadow + UBO). PCSS: 3 bindings (sampler2DArrayShadow + UBO + sampler2DArray non-compare). VSM: 2 bindings (sampler2DArray RG16F linear + UBO). `LightingPassSetShadowAlgorithm` rebuilds the pipeline *layout* on swap because the set=2 layout pointer changes, then rebuilds the pipeline with the new SPV. Already wired in Phase 10 — Phase 11 just exercised it.
* **lighting.comp lost its spec constant.** `kAlgoVariant` is gone; each variant is a single-purpose SPV. If Phase 13 wants to consolidate, it'll need to ship 4 single-variant SPVs (sdfcone added) and keep the per-variant dispatch in `Lighting.cpp`.
* **PCSS push-constant budget unchanged.** `LightingPushConstants` is still 144 bytes; the per-pixel IGN seed comes from `gl_GlobalInvocationID.xy` so no push payload growth.
* **VSM RG16F is enough.** Depth values fall in `[0, 1]` and `depth²` is `≤ 1`; RG16F's ≈ 11-bit-mantissa precision at the upper end is fine. RG32F would double the bandwidth for zero perceptible quality gain on ShaderBall — bumping to RG32F is a quality preset to remember if rough metals over distant geometry start showing variance artefacts.
* **`InitDesc::ShaderArtifactsDir` is still hard-coded inside each algo's `Initialize` as `"Artifacts/Shaders"`.** Phase 10 punted on plumbing the renderer's shader dir through; that's still TBD. Not load-bearing — every algo agrees on the same string — but it'll bite the day a host (Pigment) wants to override it.
* **No frame-time regression.** ShaderBall 3×3 grid (9 instances × 12 submeshes = 108 draws/frame) at 1280×720 stays vsync-locked (16.6 ms) on the GTX 1650 SUPER in all three variants. VSM's extra color-write moment pass is ~ 1.2 ms; PCSS's 32-tap Vogel filter is ~ 0.8 ms over PCF's 9-tap. Measured by eyeballing the `Performance Monitor` GPU graph, not by query-pool timestamps — those land in Phase 16's polish pass.
* **The picking + materials + IBL chain is unchanged.** No regression in any of the Phase 9 features when the shadow algo swaps.
* **Known limitation: floor receives no shadows.** `GridPass`'s procedural checker floor is composited onto the swapchain *after* lighting — it has no GBuffer presence, so the shadow factor never multiplies its colour. User chose to defer to Phase 14/15 (SDFGI needs the floor as a real GBuffer mesh for indirect bounces, so the fix lands together). Switching algos visibly affects ShaderBall instances only.
* **Cascade fit is duplicated** between `ShadowMap::BuildCascades` and `VSMShadow::BuildCascades`. Refactor candidate but not a Phase-11 blocker. If/when Phase 13's SDFConeShadow joins the party and *also* wants split planes (it might, for cone-radius scaling), extract `Source/Shadow/CascadeFit.{h,cpp}` and have all three impls call it.

### Phase 10 wrap-up notes
